```bash

```

**Что анализировать:**
- Топ-10 самых медленных запросов

- Количество вызовов (calls) - часто вызываемые медленные запросы особенно опасны
- Процент попаданий в кэш (hit_percent) - низкий % может указывать на недостаток индексов

## 🔧 Шаг 4: Проверка работы индексов

### 4.1 EXPLAIN ANALYZE для ключевых запросов

Подключитесь к БД и выполните:

```sql
-- Запрос 1: Получение контента с статусом покупки
EXPLAIN ANALYZE
SELECT c.*, 
       CASE WHEN p.user_id IS NOT NULL AND (pay.status = 'succeeded' OR pay.payment_id IS NULL)
            THEN true ELSE false END as has_purchased
FROM content c
LEFT JOIN purchases p ON c.id = p.content_id AND p.user_id = 123
LEFT JOIN payments pay ON p.payment_id = pay.id
WHERE c.type = 'nutrition_plan' AND cat.name = 'mass_gainer'
ORDER BY c.created_at DESC
LIMIT 20;

-- Проверьте, что используются индексы:
-- idx_content_type_category (для content.type + category_id)
-- idx_purchases_content_user (для purchases.content_id + user_id)
-- idx_payments_status_id (для payments.status)
```

**Ожидаемый результат:**
- `Index Scan` вместо `Seq Scan` (полное сканирование таблицы)
- Время выполнения <50ms для ограниченного набора данных

### 4.2 Проверка использования индекса purchases_content_user

```sql
SELECT indexname, idx_scan, idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
WHERE indexname = 'idx_purchases_content_user';

-- idx_scan должен быть > 0 (индекс используется)
-- idx_tup_fetch / idx_scan ≈ 1 (высокая эффективность)
```

## 📈 Шаг 5: Сбор метрик до/после

### 5.1 Метрики для сбора

| Метрика | Как измерить | Цель |
|---------|-------------|------|
| Время ответа на команду /start | Логи бота / ручное тестирование | <1s |
| Время загрузки категорий | Логи /performance endpoint | <200ms (из кэша) |
| Время проверки покупки | Логи /slow-queries | <100ms |
| Количество запросов к БД на пользователя | Логи, pg_stat_statements | Снизить на 60%+ |
| Использование памяти бота | `ps aux | grep bot` | Стабильное, без роста |
| Размер кэша категорий | /performance endpoint | 5-20 записей (зависит от типов контента) |

### 5.2 Сравнение с базовыми показателями

**До оптимизаций (ожидаемые проблемы):**
- 3-4 запроса на просмотр контента (N+1)
- Медленный отклик при большом количестве пользователей
- Высокая нагрузка на БД

**После оптимизаций (ожидаемые улучшения):**
- 1 запрос на просмотр контента
- Быстрый отклик даже при 1000+ пользователей
- На 60-70% меньше запросов к БД

## 🐛 Шаг 6: Выявление оставшихся проблем

### 6.1 Проверьте, что Redis работает (если используется)

```bash
# В логах бота:
# Должно быть: "FSM storage: RedisStorage" вместо "MemoryStorage"

# Проверьте подключение Redis:
redis-cli info stats | grep connected_clients
```

**Если Redis не подключен:**
- Бот будет использовать MemoryStorage
- Состояния пользователей будут теряться при перезапуске
- Производительность при большом количестве пользователей ухудшится

**Решение:** Убедитесь, что `REDIS_URL` корректно задан в `.env` и Redis доступен.

### 6.2 Проверьте, что все оптимизации активированы

**В логах бота при старте ищите:**
```
INFO - Database pool created successfully
INFO - All migrations are up to date
```

**Критические проверки:**
- [ ] Миграция 004 применена
- [ ] Индекс `idx_purchases_content_user` существует
- [ ] Кэш категорий работает (проверьте через /performance)
- [ ] Нет предупреждений "Slow query detected" в логах

### 6.3 Анализ логов бота на предмет ошибок

```bash
# Просмотр логов (если используется systemd или другой менеджер)
journalctl -u healthy-advicer-bot -f

# Или просмотр файла логов если используется файловое логирование
tail -f logs/bot.log
```

**На что обратить внимание:**
- `ERROR` - критические ошибки
